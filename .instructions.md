# Airtable-Shopify-eBay Sync App Instructions

## Project Overview

Build a real-time synchronization application that connects Airtable, Shopify, and eBay listings into a unified dashboard. The app logs all sync activities and provides visibility into listing status across all three platforms.

## Tech Stack

- **Frontend**: React 18+ with TypeScript
- **Styling**: Tailwind CSS v4
- **State Management**: React Hooks / Context API (or Redux if needed for complexity)
- **API Integration**: REST APIs for Airtable, Shopify, and eBay
- **Backend** (if needed): Node.js/Express for webhook handling and sync orchestration
- **Database**: Consider SQLite for local dev, PostgreSQL for production
- **Environment Management**: `.env.local` for API keys and configuration

## Core Features

### 1. Data Synchronization
- **Bi-directional Sync**: Push/pull data between platforms
- **Conflict Resolution**: Handle cases where listings are updated on multiple platforms
- **Scheduled Syncs**: Run sync jobs on intervals (hourly, daily, on-demand)
- **Error Handling**: Graceful degradation and retry logic

### 2. Activity Dashboard
- **Real-time Logging**: Display sync events as they occur
- **Log Filtering**: Filter by platform, date range, sync status (success/failed)
- **Activity Timeline**: Chronological view of all sync operations
- **Statistics Panel**: Count of items synced, success rate, last sync time per platform

### 3. Listing Management
- **Inventory View**: Display all listings across platforms
- **Listing Details**: Show current status, price, quantity on each platform
- **Manual Sync Trigger**: Allow users to manually trigger sync for specific listings
- **Mapping Management**: Define which Airtable records map to Shopify/eBay products

### 4. Settings & Configuration
- **API Key Management**: Securely store and manage API credentials
- **Sync Rules**: Configure which fields sync between platforms
- **Webhook Configuration**: Set up webhooks for real-time updates
- **Notifications**: Alert settings for failed syncs or conflicts

## Key Components

### Dashboard Page
- Central hub showing sync status, recent activity, and statistics
- Quick access buttons to trigger manual syncs
- Real-time updates using WebSocket or polling

### Activity Log
- Paginated table of all sync events
- Columns: Timestamp, Platform, Action, Items Processed, Status, Details
- Click-through to see error messages or details
- Export functionality (CSV)

### Listings Grid
- Cards or table view of all products
- Show status (in-sync, conflict, needs-review, error)
- Visual indicators for platform availability
- Search and filter capabilities

### Settings Panel
- Secure credential management for API keys
- Test connection buttons for each platform
- Sync interval configuration
- Field mapping UI for custom sync rules

## Development Workflow

### Setup
1. Clone repository and install dependencies
2. Create `.env.local` with API keys for Airtable, Shopify, eBay
3. Set up database (SQLite for dev or PostgreSQL)
4. Run migrations if applicable
5. Start development server: `npm run dev`

### API Integration Steps
1. **Airtable**: Authenticate using API token, fetch base/table structure
2. **Shopify**: Set up custom app, store access token
3. **eBay**: Configure OAuth or use application-level API keys

### Local Development
- Use Tailwind CLI or Next.js/Vite built-in support for styling
- Mock API responses for testing without live credentials
- Use React DevTools for component debugging
- Set up ESLint and Prettier for code quality

### Deployment Considerations
- Encrypt API keys in environment variables
- Use HTTPS for all API communications
- Implement rate limiting awareness (each platform has limits)
- Consider background job queues for heavy sync operations
- Set up monitoring and alerting for failed syncs

## UI/UX Guidelines (Tailwind v4)

- **Tailwind-Only Component Styling**: When creating or updating React components, style with Tailwind CSS v4 utility classes directly in the component. Do not introduce or extend component-specific CSS class blocks in `App.css` (or other global CSS files) unless there is a clear cross-cutting global style requirement.
- **Color Scheme**: Use consistent palette for status indicators (green=success, red=error, yellow=warning)
- **Responsive Design**: Mobile-first approach, breakpoints for tablet/desktop
- **Accessibility**: ARIA labels, semantic HTML, keyboard navigation
- **Loading States**: Skeleton screens and spinners for async operations
- **Error Boundaries**: Wrap major sections to prevent full app crashes

## Important Considerations

- **Rate Limiting**: Each platform has API rate limits; implement exponential backoff
- **Data Validation**: Validate data before pushing to platforms
- **Audit Trail**: Keep comprehensive logs of all sync operations and conflicts
- **Testing**: Unit tests for sync logic, integration tests for platform connections
- **Security**: Never expose API keys in client-side code; use backend for credential management
- **Performance**: Consider caching strategy, optimize re-renders with React.memo and useMemo
